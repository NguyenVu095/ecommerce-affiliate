## I. TIÊU CHUẨN LẬP TRÌNH (CODING STANDARDS)

Để đảm bảo toàn bộ dự án duy trì được hiệu suất cao, tính dễ đọc và bảo trì lâu dài, tất cả các tác vụ refactor hoặc viết mới code cần phải tuân thủ nghiêm ngặt các tiêu chuẩn sau:

### 1. Backend (FastAPI / SQLAlchemy / Python)
*   **Quy tắc đặt tên:**
    *   **Thư mục & File:** Luôn đặt dạng `snake_case` (ví dụ: `product_variant.py`, `order_status.py`).
    *   **Class/Model:** Luôn đặt dạng `PascalCase` (ví dụ: `ProductVariant`, `OrderAddress`).
    *   **Hàm & Biến:** Luôn đặt dạng `snake_case` (ví dụ: `get_active_coupons`, `current_user`).
    *   **Hằng số:** Dạng `UPPER_CASE` viết trong các file core/config (ví dụ: `ACCESS_TOKEN_EXPIRE_MINUTES`).
*   **Tính dễ đọc & Cấu trúc:**
    *   **Type Hinting:** Bắt buộc định nghĩa kiểu dữ liệu đầu vào và đầu ra cho tất cả các hàm (ví dụ: `def get_user(db: Session, user_id: int) -> User | None:`).
    *   **Docstring:** Viết mô tả ngắn ở đầu mỗi class phức tạp và API route quan trọng giải thích rõ luồng xử lý hoặc nghiệp vụ đặc thù.
    *   **Xử lý lỗi:** Không sử dụng các khối `except Exception: pass` trống. Phải ném lỗi rõ ràng bằng `HTTPException` kèm mã lỗi thích hợp (`400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`).
*   **Hiệu năng & Tối ưu hóa Thuật toán:**
    *   **DB Transactions (Tối ưu hóa Giao dịch):** Hạn chế tối đa việc gọi `db.commit()` liên tục trong cùng một API. Mỗi lần commit sẽ kích hoạt ghi đĩa vật lý đồng bộ (Disk I/O) rất chậm. Luôn ưu tiên dùng `db.flush()` để đồng bộ hóa trạng thái tạm thời và sinh ID tự động, sau đó chỉ gọi `db.commit()` **duy nhất 1 lần** ở cuối luồng xử lý thành công. điều này vừa tăng tốc độ phản hồi API vừa đảm bảo tính nguyên tố ACID (hoặc cùng thành công hoặc rollback toàn bộ khi có lỗi).
    *   **Tránh lỗi N+1 Query:** Khi truy vấn các bảng có mối quan hệ (relationship), bắt buộc sử dụng `joinedload` hoặc `selectinload` của SQLAlchemy để tải trước dữ liệu liên kết, không truy cập Lazy Load trong vòng lặp.
    *   **Deduplicate Từ khóa:** Khi xây dựng các câu truy vấn động dựa trên đầu vào của người dùng (ví dụ: tìm kiếm sản phẩm), phải thực hiện lọc bỏ từ khóa trùng lặp trước để tránh tạo ra các điều kiện `ILIKE` lặp lại dư thừa làm nặng bộ phân tích SQL.
    *   **Logging:** Nghiêm cấm dùng `print()` để log trong API route hoặc các hàm xử lý. Phải sử dụng module `logging` chuẩn của Python (ví dụ: `logger.info()`, `logger.error()`) để dễ dàng giám sát hệ thống.

### 2. Frontend (React / TypeScript / Vite)
*   **Quy tắc đặt tên:**
    *   **Components:** Luôn đặt tên theo `PascalCase` cho cả tên hàm và tên file `.tsx` (ví dụ: `ProductCard.tsx`, `CartSummary.tsx`).
    *   **Hooks:** Bắt đầu bằng tiền tố `use` theo sau là `camelCase` (ví dụ: `useCart.ts`, `useAuth.ts`).
    *   **Biến & Hàm:** Luôn sử dụng `camelCase` (ví dụ: `totalAmount`, `handleCheckCoupon`).
*   **Độ an toàn kiểu dữ liệu (Strict Type Safety):**
    *   Nghiêm cấm lạm dụng kiểu dữ liệu `any`. Bắt buộc khai báo interface/type tường minh cho mọi props, state, API response.
    *   Nếu bắt buộc phải dùng `any` do thư viện bên ngoài không hỗ trợ type, phải viết comment giải thích cụ thể lý do tại dòng đó.
*   **Hiệu năng & Trải nghiệm người dùng:**
    *   **Tối ưu hóa Vòng lặp Render (Tần suất quét mảng):** Tránh việc gọi các hàm quét mảng $O(N)$ như `.filter()`, `.find()`, `.map()` liên tục lặp lại bên trong vòng lặp render chính (ví dụ: lặp qua các Tab bộ lọc rồi gọi filter đơn hàng). Phải tính toán trước dữ liệu đếm hoặc bản đồ tần suất bằng `useMemo` trong duy nhất **1 vòng quét** $O(N)$, sau đó lấy dữ liệu đếm bằng truy cập key $O(1)$.
    *   **Lazy Loading:** Sử dụng `React.lazy()` và `Suspense` cho tất cả các Route lớn để tối ưu kích thước gói bundle tải ban đầu (đã áp dụng trong `App.tsx` của `frontend_user`).
    *   **Tránh Re-render thừa:** Sử dụng `useMemo` cho các phép toán tính toán nặng và `useCallback` khi truyền hàm qua nhiều cấp component con. Bắt buộc truyền `key` duy nhất và ổn định khi map mảng (không dùng chỉ số index mảng làm key nếu mảng đó có thể thay đổi thứ tự/xóa bớt).
    *   **Quản lý Axios / Fetch:** Tất cả các lệnh gọi API phải được bọc trong thư mục `services/` tập trung, có xử lý catch lỗi đầy đủ và hiển thị thông báo thân thiện (toast/alert) tới người dùng. Không được viết `fetch()` trực tiếp dạng inline trong UI component.
    *   **Dọn dẹp log:** Xóa hoàn toàn các dòng `console.log()` trước khi đẩy code lên môi trường production. Chỉ giữ lại các log dạng `console.error()` trong các khối catch xử lý ngoại lệ quan trọng.

## II. TIÊU CHÍ DỌN DẸP MÃ NGUỒN (CLEANUP PROTOCOL)

Hành vi dọn dẹp mã nguồn phải thực hiện theo các bước chuẩn mực để tránh xóa nhầm các đoạn code tính năng hoặc làm hỏng cấu trúc dự án:

1.  **Nhận diện Code rác (Dead Code & Junk Files):**
    *   Các biến, hàm, component hoặc các dòng `import` được khai báo nhưng không bao giờ được gọi đến (sử dụng ESLint trên Frontend và Flake8/Autoflake trên Backend để phát hiện).
    *   Các khối code cũ bị tắt hoạt động bằng cách chuyển thành comment thay vì bị xóa hẳn.
    *   Các file nháp tạm thời, file kết quả test lỗi, hoặc công cụ mock tự chế không có giá trị hoạt động trên Production (ví dụ: các file log cũ, các script kiểm thử tải đặt sai chỗ).
    *   Các thư mục trống rỗng được tạo ra nhưng không chứa bất kỳ file mã nguồn nào.
2.  **Quy trình Refactor:**
    *   *Bước 1:* Phân tích xem đoạn code hoặc file đó có thực sự vô dụng hay không (kiểm tra toàn bộ các tham chiếu chéo trong dự án).
    *   *Bước 2:* Tiến hành xóa bỏ hoặc tối ưu hóa thuật toán bên trong.
    *   *Bước 3:* Chạy thử ứng dụng (Build Frontend, Khởi chạy Backend) để đảm bảo không xảy ra lỗi thiếu import hoặc sập hệ thống.
    *   *Bước 4:* Ghi nhận lại chính xác những gì đã thay đổi vào **Bảng nhật ký tiến độ** bên dưới để phiên chat AI tiếp theo không làm trùng lặp.

## III. QUY TẮC BẮT BUỘC KHI TỐI ƯU HÓA HÀM CODE
*(Mọi hoạt động tối ưu hóa hiệu năng, thuật toán cho các hàm xử lý dữ liệu và truy vấn cơ sở dữ liệu trong toàn bộ dự án bắt buộc phải tuân thủ nghiêm ngặt các nguyên tắc cốt lõi sau)*

1.  **GIỮ NGUYÊN TÍNH NĂNG NGHIỆP VỤ:** Tuyệt đối không thay đổi hành vi nghiệp vụ hiện tại của hệ thống. Đầu ra và phản hồi của code sau khi tối ưu phải khớp chính xác 100% với phiên bản cũ dưới mọi điều kiện hoạt động.
2.  **GIỮ NGUYÊN BƯỚC KIỂM TRA DỮ LIỆU ĐẦU VÀO (VALIDATION):** Giữ nguyên toàn bộ các bước kiểm tra, ràng buộc, xác thực dữ liệu đầu vào (như kiểm tra rỗng, kiểm tra kiểu dữ liệu, các tầng phân quyền, kiểm tra lỗi hợp lệ). Không được lược bỏ hay làm lỏng lẻo các bước kiểm tra bảo mật này nhằm đổi lấy hiệu suất.
3.  **MÃ NGUỒN DỄ ĐỌC VÀ DỄ BẢO TRÌ:** Code mới sau khi tối ưu phải có cấu trúc mạch lạc, rõ ràng, sử dụng các biến có tên mang tính tự giải thích. Tránh viết code tối giản quá mức (clever code) gây khó khăn cho việc bảo trì.
4.  **GIẢI THÍCH CHI TIẾT ƯU ĐIỂM THUẬT TOÁN MỚI:** Trong các file code được refactor, bắt buộc phải viết comment rõ ràng giải thích:
    *   Thuật toán hoặc cách tiếp cận mới là gì.
    *   Điểm tối ưu vượt trội của thuật toán mới so với thuật toán cũ (ví dụ: giảm độ phức tạp thời gian từ $O(K \times N)$ xuống $O(N)$, hoặc giảm số lần Disk I/O từ 4 lần xuống 1 lần).
5.  **TRÌNH TỰ ƯU TIÊN TỪ DƯỚI LÊN TRÊN (BOTTOM-UP PRIORITY):** Việc rà soát và tối ưu hóa các thành phần bắt buộc phải thực hiện theo đúng trình tự từ dưới lên trên dựa theo **Danh sách kiểm tra dự án** (ở mục IV bên dưới) - đi từ các module ở đáy danh sách (ví dụ: Frontend User `account` -> Backend `chat`) dần lên các module ở phía trên.
